# v0.9.0 MVP Backlog

## Goal

Ship a branded INDIGO Studio foundation release that is ready for repeated local, preview, and production operations without changing the runtime product shape yet.

This release is the control-plane baseline for:

- repo, git, env, deploy, and validation workflows
- Strapi-first CMS starter variations
- composable blueprints for future modules
- multibrand product launch operations
- revenueOS dashboards and launch support

## Non-Goals

- no new app architecture rewrite
- no new feature-flag service
- no new provider platform unless a current surface cannot cover the need
- no implementation of the docs site beyond a placeholder backlog entry
- no attempt to solve every backlog idea in one release

## Release Lock

v0.9.0 is locked when all of the following are true:

- the repo bootstrap is deterministic on a clean machine
- local, preview, and prod env surfaces are documented and reproducible
- Easypanel and Vercel paths are validated from the repo command surface
- the Strapi admin flow is stable enough for onboarding and existing admin reuse
- `/dashboard` is treated as the launchops control surface, not a product feature detour
- no tracked config file contains live secrets
- the release is split into GHP-sized issues and worktrees

## Release Tracks

### Urgent

These are the only items that block a v0.9.0 foundation release:

- normalize repo, env, and deployment commands into one deterministic path
- keep the live Easypanel source pointed at the Git-backed repo, not stale inline YAML
- keep the internal Strapi compose service name stable
- validate the Strapi Dockerfile and compose contract against the repo-root build context
- keep the admin path and public URL contract explicit
- preserve the existing admin database when migrating an active environment
- keep secrets out of tracked env files and scripts

### Next

These are the first follow-on slices after the foundation is stable:

- branded dashboard shell for `/dashboard`
- deploy status, health, and control panels in one visual surface
- feature-flag scaffold using existing env/content switches only
- sample MVP content, collections, and resources
- reusable module blueprints for product launches and revenue operations
- local and preview data/bootstrap paths for demoable starter content

### Backlog

These items are intentionally deferred until the foundation proves durable:

- docs site TBD
- broader provider adapters beyond the current repo surfaces
- deeper agent-team automation beyond issue/worktree/PR flow
- richer launch analytics and perf telemetry
- long-form design system expansion
- custom workflow engines or extra platform services
